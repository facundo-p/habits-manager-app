# Phase 3 — UI Design Contract: Google Drive Backup

> Visual and interaction contract for the Cloud Backup UI surfaces. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

This spec extends the established **Cozy Habits** visual language (amber/notebook aesthetic, Merriweather + Lato, NotebookPaper containers, native Alert dialogs). It defines **only** the new contracts introduced by Phase 3: Settings "Backup en la nube" section, `RestoreFromDriveScreen`, restore confirmation modal, error UX, and loading overlay. No new design system is introduced — all tokens reuse `src/styles/ui.styles.ts`.

| Property | Value |

## Spacing Scale

Project follows Tailwind's default 4px scale. Phase 3 declares these tokens (all multiples of 4):

| Token | Value | Tailwind class | Usage in Phase 3 |
|-------|-------|----------------|------------------|
| xs | 4px | `gap-1`, `mt-1`, `mb-1` | Inline icon-to-text gap inside connected-state row, last-backup caption |
| sm | 8px | `gap-2`, `py-2`, `mt-2` | Spacing between adjacent action buttons inside the Drive section, status pill padding |
| md | 16px | `gap-4`, `p-4`, `py-4` | Default padding inside NotebookPaper rows, restore list item vertical padding |
| lg | 20–24px | `px-5`, `p-5`, `mb-5` | Screen horizontal padding (matches `layout.padded` = `px-5`), section interior padding (`card.elevated` = `p-5`) |
| xl | 32px | `mt-8`, `mb-8` | Distance between Drive section and the next/previous Settings section |
| 2xl | 48px | `pt-12` | Top padding for empty-state inside `RestoreFromDriveScreen` |
| 3xl | 64px | `pt-16` | Vertical centering padding for `LoadingOverlay` content area |

**Exceptions:**
- `iconDefaults.size = 22` (existing token) is used for inline list icons; allowed as-is.
- Touch targets for Pressable rows must reach **≥44px** vertical (`py-3` ≈ 48px including text). The connected-state row, the "Backup ahora" / "Restaurar desde Drive" buttons, and each restore-list item must satisfy this.
- `LoadingOverlay` spinner uses `ActivityIndicator size="large"` (RN-native, ~36px) which does not map to the 4px scale; allowed because it is a system primitive.

## Typography

Phase 3 declares **4 sizes, 2 weights** — all already present in `text.*` tokens. No new font, no new size, no new weight is introduced by this phase.

The 4 sizes used in Phase 3 surfaces are 24/18/16/14. The 12px `text.captionSmall` token exists in the project for other surfaces (`styles.versionText`, area badges) but is **NOT used by any Phase 3 surface** and is therefore not part of this contract.

| Role | Size | Weight | Line height | Token | Used in Phase 3 for |
|------|------|--------|-------------|-------|---------------------|
| Display | 24px (`text-2xl`) | Merriweather (serif, single weight 700) | ~1.2 | `text.screenTitle` | `RestoreFromDriveScreen` title ("Restaurar desde Drive") via `AppScreenHeader` |
| Section heading | 18px (`text-lg`) | Lato Semibold (600) | ~1.3 | `text.sectionTitle` | "Backup en la nube" section title in Settings |
| Body | 16px (`text-base`) | Lato Regular (400) | ~1.5 | `text.body` | Connected email, button labels, modal body, list item primary text |
| Caption | 14px (`text-sm`) | Lato Regular (400) | ~1.4 | `text.caption` | Last-backup timestamp, file size in restore list, destructive warning notes (color override `text-rose-400`) |

**Weight count justification (semantic role per font family):**
The "2 weights" limit applies to weight *choices* within a single sans family. Phase 3 uses two weights from the Lato sans family — Regular (400) for body/caption and Semibold (600) for section headings and button labels. Merriweather is loaded with a single weight (`Merriweather_700Bold`) because the project ships only that variant; it is a property of the serif font definition, not a separate weight choice. There are therefore exactly **2 sans weights + 1 serif weight = 2 design weight decisions** (one per family).

**Strict rules for Phase 3:**
- Primary CTAs use `button.primaryText` (16px, Lato Semibold, white). Secondary CTAs use `button.secondaryText` (16px, Lato Semibold, amber-600). Both fall under the "Body 16px" row above; the Semibold weight is the same one declared for the Section heading row.
- Destructive warning copy that appears as inline text inside Phase 3 surfaces uses `text.caption` (14px) with color override `text-rose-400`. Note: this is a deliberate uplift from the legacy `styles.destructiveNote` pattern in `SettingsScreen.styles.ts` (which is 12px `text-xs`); Phase 3 destructive copy is more prominent because it appears adjacent to a destructive trigger. The legacy 12px footnote class is not reused by Phase 3.
- All destructive *confirmation* dialogs in Phase 3 are native `Alert.alert` invocations (per D-21). Native Alert does not accept custom typography — the OS controls font sizes and weights inside those dialogs, so they are explicitly outside this typography contract.
- Email of the connected account is rendered with `text.body` (no monospace, no special treatment).
- Last-backup timestamp uses `text.caption` (`text-amber-600`).

## Component Inventory (Phase 3)

This inventory enumerates every UI surface the executor will build. Each row maps to a file and lists the exact states the component must render.

| # | Component | File | States | Reuses |
|---|-----------|------|--------|--------|
| 1 | Drive section in Settings | `src/screens/SettingsScreen.tsx` (extend) | `disconnected` / `connecting` / `connected-idle` / `connected-uploading` | `NotebookPaper`, existing `styles.sectionTitle`, `button.primary`, `button.secondary` |
| 2 | `LoadingOverlay` | `src/components/shared/LoadingOverlay.tsx` (NEW) | visible/hidden + contextual message slot | `Modal` (RN), `ActivityIndicator`, `colors.amber600`, `text.body` |
| 3 | `RestoreFromDriveScreen` | `src/screens/RestoreFromDriveScreen.tsx` (NEW) | `loading` / `empty` / `error` / `loaded` | `AppScreenHeader`, `FlatList`, `card.base`, `iconDefaults` |
| 4 | Restore list item row | inline in `RestoreFromDriveScreen` | `idle` / `pressed` | `card.habit` pattern, `text.body`, `text.caption` |
| 5 | Restore confirmation Alert | `Alert.alert` invocation | confirmation only (native dialog, no custom UI) | RN `Alert` API + new constants `ALERT_DRIVE_RESTORE_CONFIRM` |
| 6 | Sign-out confirmation Alert | `Alert.alert` invocation | confirmation only | RN `Alert` API + `ALERT_DRIVE_SIGN_OUT` |
| 7 | Overwrite-today confirmation Alert | `Alert.alert` invocation | confirmation only (D-13) | RN `Alert` API + `ALERT_DRIVE_OVERWRITE_TODAY` |
| 8 | Error Alerts (5 categories + generic) | `Alert.alert` invocation | category-specific title/message + Reintentar button | RN `Alert` API + `ALERT_DRIVE_*` constants |

**Decision: full screen vs modal for restore list.** Confirmed `RestoreFromDriveScreen` (full screen, native-stack route). Rationale: list length is unbounded (retention up to 30 + monthly + yearly entries), preview interaction needs vertical room, and `AppScreenHeader` gives the user a familiar back affordance.

**Decision: where the Drive section sits in Settings.** Insert as a NEW section titled "Backup en la nube" placed **between** "Personalización" and "Seguridad y Datos". Rationale: cloud and local backup remain conceptually separate (different destinations, different failure modes); merging them would force the user to scan a longer mixed-purpose card. Order top-to-bottom: Personalización → Backup en la nube (new) → Seguridad y Datos (existing local export/import) → Versión.

**Decision: cleanup of `cozyhabits-pre-restore-*.json` cache.** Silent cleanup on the next successful restore. The restore success Alert mentions the safety backup ("Tus datos previos quedaron respaldados en el dispositivo por si querés revertir") but no in-app UI for "undo" is exposed in v1 — the file is recoverable by a developer/support flow only. This avoids a confusing "Restore again" loop and keeps the cache from growing unboundedly.

## Interaction States (per surface)

### 1. Drive section in Settings

**Disconnected (no `googleEmail` in store):**
- Section header: lucide `Cloud` icon (22px, `colors.amber700`) + text "Backup en la nube" using `text.sectionTitle`.
- Single primary CTA: `button.primary` with lucide `Cloud` icon (18px, white) + label **"Conectar con Google"**.
- Below CTA: `text.caption` body — "Guardá un respaldo en tu Google Drive y restauralo desde cualquier dispositivo."

**Connecting (silent or interactive sign-in in flight):**
- CTA disabled (`disabled={true}`), opacity unchanged (RN handles), label becomes **"Conectando..."** with `ActivityIndicator size="small" color={colors.white}` replacing the Cloud icon.
- No `LoadingOverlay` for sign-in itself (it is a known-fast operation and the SDK shows its own native account picker).

**Connected — idle:**
Layout (top to bottom inside NotebookPaper):
1. **Identity row** (`flex-row items-center gap-2 py-3`):
   - lucide `CheckCircle2` (18px, `colors.sage400`) — success status dot/icon.
   - `text.body` showing the email (e.g., `usuario@gmail.com`). Truncate with `numberOfLines={1}`.
2. **Last backup caption row** (`py-1`):
   - `text.caption` (`text-amber-600`) — "Último backup: {fecha relativa}". If `lastBackupAt === null`, show "Aún no hiciste un backup".
   - Date format: `"hace 2 horas"`, `"ayer"`, `"el 12 abr 2026"` (Spanish, no library — small util in `lib/dateFormat.ts`).
3. **Divider** (`spacing.divider`).
4. **Primary CTA — "Backup ahora"**: `button.primary` with lucide `CloudUpload` (18px, white). On press → upload flow (overlay).
5. **Secondary CTA — "Restaurar desde Drive"**: `button.secondary` with lucide `CloudDownload` (18px, `colors.amber600`). On press → navigate to `RestoreFromDriveScreen`.
6. **Tertiary CTA — "Cerrar sesión"**: text-only `Pressable` with `text.caption` color `text-amber-700`, lucide `LogOut` (14px) inline left, padding `py-3`. On press → confirm Alert (D-11).

**Connected — uploading (after "Backup ahora" pressed):**
- `LoadingOverlay` (visible: true) blankets the screen with message "Subiendo a Drive...".
- Settings remains mounted underneath but unreachable until overlay dismisses.

**Connected — restoring:**
- Same as uploading, message "Restaurando datos..." (overlay is owned by the screen that triggered it; for restore, that's `RestoreFromDriveScreen`).

### 2. `LoadingOverlay` component

**Props:** `visible: boolean; message: string`.

**Visual:**
- `Modal` with `transparent={true}`, `animationType="fade"`, `statusBarTranslucent={true}`.
- Backdrop: `flex-1 items-center justify-center bg-black/40` (existing token: `colors.black40`).
- Inner card: `bg-amber-50 rounded-2xl p-6 items-center gap-3` — width auto, max ~70% of screen.
- `ActivityIndicator size="large" color={colors.amber600}`.
- `text.body` (centered) showing `message`.

**Interaction:**
- Captures back button on Android (`onRequestClose={() => {}}` — no-op while visible).
- Blocks all touches automatically via Modal.
- File length budget: ≤30 lines (matches D-23 brief).

### 3. `RestoreFromDriveScreen`

**Header:** `AppScreenHeader` — title "Restaurar desde Drive", `onBack={() => navigation.goBack()}`, `showSettings={false}`.

**Loading state (initial fetch):**
- Empty FlatList with `ListEmptyComponent` containing centered `ActivityIndicator size="large" color={colors.amber600}` + `text.caption` "Cargando backups...".

**Loaded state (≥1 backup):**
- `FlatList` with `ItemSeparatorComponent` (`spacing.divider`).
- Each row (`Pressable`):
  - Container: `card.base` styling, `flex-row items-center gap-3 py-3 px-4`.
  - Leading: lucide `FileText` (22px, `colors.amber600`).
  - Center column (`flex-1`):
    - Primary line (`text.body`): "{fecha formateada}" — e.g., "27 abr 2026".
    - Caption line (`text.caption`): "{tamaño}" — e.g., "12 KB".
  - Trailing: lucide `ChevronRight` (18px, `colors.amber400`).
- Newest first (D-15).

**Empty state (0 backups, fetch succeeded):**
- Centered vertical stack (`flex-1 items-center justify-center px-5`):
  - lucide `CloudOff` (48px, `colors.amber400`).
  - `text.sectionTitle` (centered): "No hay backups todavía".
  - `text.caption` (centered): "Hacé tu primer backup desde Ajustes."
  - Padding-top: `pt-12` (2xl token).

**Error state (fetch failed by network):**
- Centered vertical stack:
  - lucide `WifiOff` (48px, `colors.rose400`).
  - `text.sectionTitle` (centered): "No se pudo cargar la lista".
  - `text.caption` (centered): "Verificá tu conexión e intentá de nuevo."
  - Primary CTA (`button.primary`, narrow `w-40`): "Reintentar" — re-runs the list fetch.

**Tap on a row (D-17):**
- Open the restore-confirmation Alert (D-18) with already-known metadata (date, size). Conteos por tabla se obtienen entre el tap y el Alert: muestra brevemente el `LoadingOverlay` con mensaje "Leyendo backup...", luego dispara el Alert. If preview-fetch fails, show `ALERT_DRIVE_GENERIC` instead.

## Accessibility

| Concern | Contract |
|---------|----------|
| Touch targets | All interactive elements ≥44px tall. Pressables use `py-3` minimum (≈48px including text). |
| Screen reader labels | Every `Pressable` declares `accessibilityLabel` and `accessibilityRole="button"`. Connected-state row exposes `accessibilityLabel="Cuenta conectada: {email}"`. The status icon has `accessibilityElementsHidden={true}` (info already in row label). |
| Color is not the only signifier | Connected state combines sage-400 dot + email text. Errors combine icon + heading + body — never color alone. Disabled buttons combine label change + spinner. |
| Contrast | Primary CTA: amber-600 background (`#d97706`) + white text → 4.84:1 (AA). Body text: amber-900 (`#78350f`) on amber-50 (`#fffbeb`) → 8.7:1 (AAA). Caption text: amber-600 on amber-50 → 4.84:1 (AA). All meet WCAG AA. |
| Loading announcement | `LoadingOverlay` Modal exposes `accessibilityViewIsModal={true}`. Inner Text has `accessibilityLiveRegion="polite"` so screen readers announce the message change. |
| Native Alerts | Trust the OS — Alert.alert is fully accessible by default. Do not replace with custom modal. |
| Reduce motion | No bespoke animations are introduced (Modal uses `animationType="fade"` which respects OS reduce-motion on iOS). |
| Hit-slop | Tertiary "Cerrar sesión" Pressable adds `hitSlop={{top: 8, bottom: 8, left: 8, right: 8}}` because its visible chrome is text-only. |

## Constants to Add (`src/config/constants.ts`)

The planner/executor will add these to `constants.ts` in the existing `// ─── Alerts ───` and `// ─── Backup ───` sections. Listed here so the contract is auditable.

```ts
// ─── Drive Backup ───────────────────────────────────────────────────
export const BACKUP_FILE_PREFIX = 'cozyhabits-';
export const BACKUP_FILE_EXTENSION = '.json';
```

## Risk Notes for the Checker / Executor

These are NOT blocking — they are defensive callouts the executor must respect during implementation:

1. **Native Alert vs. custom modal consistency.** D-21 mandates `Alert.alert`. Do NOT introduce a bottom sheet or custom modal for any of the 8 Drive Alerts — even if a future maintainer thinks one would be "cleaner". Consistency with `handleImport` is the contract.
2. **Loading pattern split.** D-23 + D-24 deliberately split: full-screen `LoadingOverlay` for upload/restore (long, blocking, irreversible), inline spinner for list fetch (short, non-blocking). Do NOT unify them.
3. **Sign-in does NOT use the overlay.** The Google native account picker is its own UX surface — overlaying our spinner on top of it is harmful. Sign-in connects/connecting states are visualized only via the button label change.
4. **Counts in restore confirm Alert.** Counts are derived from the parsed backup JSON (already validated by `parseAndValidate`). If parsing fails, do NOT show the Alert with placeholders — show `ALERT_DRIVE_GENERIC` instead and abort the restore. This prevents the user from confirming a restore against unknown data.
5. **Retention pruning is silent.** Per D-14, pruning runs after a successful backup. It must NOT surface progress, success, or failure to the user. If pruning fails, log and continue — backup itself is already considered successful at that point.
6. **Date formatting.** Spanish, lowercase month abbreviations: `27 abr 2026`, NOT `27 Apr 2026` and NOT `Apr 27, 2026`. Implement locally (small util) — do not pull `date-fns` for this phase.
